What this tool does

It is a quality-control checker for the journey from a signed deposit to an issued contract. It reads the documents, compares what was signed against what the contract output says, and gives you an itemised list of anything that doesn't match — so nothing is missed before a contract goes to the client.

It runs in three stages. Each stage is a separate tab at the top of the screen.

StageWhen you use itWhat it checks
Stage 1: PSEAt deposit timeThe original deposit submission QA (unchanged from before). Upload the deal-code zip, get a verdict and corrected zip.
Stage 2: NHPAfter estimating produces the NHPConfirms every signed variation (VO) made it into the Final NHP pricing, with matching amounts and a total that reconciles.
Stage 3: ContractBefore the contract is issuedTwo-pass check that the full contract pack (spec, pricing, drawings) matches the signed source. Itemised by spec section and drawing page.

The golden rule: it never guesses

This tool is built to be cautious. If something is unclear, it does not pretend to know — it flags it for a human to check. That means:

  • It would rather miss a borderline item than raise a false alarm.
  • If a document can't be read (e.g. a scanned image with no text), it tells you to review manually instead of passing it.
  • Stage 3 drawing findings are always marked "Confirm on drawing" — the tool points you to the sheet, but a person makes the final call.

Why this matters: a false positive wastes estimating and drafting time and erodes trust in the tool. A flagged "please check" item costs you 30 seconds. So the tool is deliberately tuned to flag-don't-assert.

Stage 1 — PSE / Deposit QA

Unchanged. Drop the deal-code-named .zip (e.g. S26TLS.zip) into the Stage 1 tab and click Run QA Review. You get the same verdict, corrected zip, and Heath note as before.

Stage 2 — NHP Review

What you upload

  1. NHP Changes document — the PDF listing the variation orders (additions, deletions, credits, debits).
  2. Final NHP PDF — the pricing document that forms the contract price.

What you get back

  • A count of VOs found, how many matched, how many are missing, and how many have a wrong amount.
  • A total reconciliation: base price + the net of all signed VOs should equal the Final NHP total. If it's out, you see the exact dollar gap.
  • An itemised list of every discrepancy, with the signed source, what the NHP says, and the required action.

Verdicts

VerdictMeaning
PASSEvery signed VO carried through with matching amounts and a reconciling total.
REVIEW REQUIREDAmounts don't match, the total doesn't reconcile, or some items need a human to confirm.
FAILOne or more signed VOs appear to be missing from the Final NHP. Do not proceed until estimating resolves it.

The two passes

Pass 1 (automatic): compares the signed source against the contract spec and pricing text — VO carry-through, debit/credit matching, credit traceability, deleted items still listed, pricing-vs-spec contradictions, and metadata (names, lot, facade, etc.).

Pass 2 (AI-assisted drawing review): reads the working-drawing pages and points you to likely issues on each sheet — window/door heights, fixture positions, deleted items still drawn, electrical fixture type, clearances. Every Pass 2 item must be confirmed by you on the actual drawing.

How results are laid out

Issues are grouped by section, matching the QA template Lyana uses — by spec item (e.g. Item 7 Facade) and by drawing page (e.g. Elevations, Ensuite). Each issue shows severity, category, what was signed, what the contract says, the problem, and the required action.

Severity

Critical contract must not be issued   High fix before issue   Medium resolve where practical   Low minor   Observation note to monitor

Marking issues (what each button does)

On any Stage 2/3 result, each issue has action buttons:

  • Fixed — the discrepancy has been corrected in the documents.
  • Accept Exception — a manager has agreed to proceed despite the issue (record the reason). Remember: Josh's call overrides.
  • False Positive — the tool was wrong; this isn't a real issue. This feeds the Learning panel.
  • N/A — not applicable to this job.

Rules & Learning — making the tool smarter without a developer

This is the important part: you can improve the tool yourself. No code, no developer, no waiting.

Rules tab

Every check the tool makes comes from a rule in the Rule Library. With an admin code you can:

  • Change a severity — e.g. drop something from High to Medium if it's over-flagging.
  • Switch a rule off — if a check isn't useful, deactivate it.
  • Add an exclusion — tell a rule what it should not flag (e.g. "gas cooktops are allowed in all estates"). The exclusion is fed straight into the AI on the next review.
  • Add a brand-new rule — describe a new check in plain English and pick its stage, category, and severity.

Learning tab

This lists everything staff have recently marked as False Positive or N/A. It's your shortlist of what to fix: if the same kind of false positive keeps appearing, go to the Rules tab and add an exclusion or lower the severity. Over a few weeks this is how the tool stops repeating mistakes.

Changes to rules apply to the next review, not past ones. Past reviews stay as a permanent record.

Quick FAQ

The tool says REVIEW REQUIRED but I can't see a dollar problem — why?

It often means an amount or the NHP total couldn't be read with confidence, or a VO matched on amount but the description match was weak. That's the tool being honest rather than guessing. Open the itemised list — each item tells you exactly what to confirm.

Stage 3 parked my job. What now?

One of the four required documents is missing (signed source, contract spec, contract pricing, or working drawings). Add it and re-run.

A drawing issue looks wrong.

Pass 2 items are pointers, not verdicts. If it's wrong, mark it False Positive — that's expected and it helps the Learning panel.

Who can edit rules?

Anyone with an admin access code. Everyone else can run reviews and mark issue status.
